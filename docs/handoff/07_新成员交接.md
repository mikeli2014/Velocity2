# 07 · 新成员交接文档

欢迎接手 Velocity OS。本文档假设你是一个有经验的全栈工程师(熟悉 React + FastAPI + 一些 LLM 应用),目标是让你在 1-2 个工作日内能独立修改代码、调试问题、上线一个小功能。

## 第 1 天 · 环境准备

### 1.1 拉代码

```bash
git clone <repo-url> velocity2
cd velocity2
```

主分支 `main`,所有功能已合并。功能分支命名规范:`feat/...` 或 `fix/...`。

### 1.2 前端开发环境

```bash
node --version    # 需要 >= 20
npm install       # 一次性,不要用 yarn / pnpm
npm run dev       # 启动 Vite,默认 http://localhost:5173
```

Vite 配置了 `/api/*` 代理到 `localhost:8081`,所以要同时启动后端。

### 1.3 后端开发环境

```bash
cd apps/api
python --version  # 3.11+
python -m venv .venv && source .venv/bin/activate
pip install -r requirements-dev.txt
uvicorn velocity_api.app:app --reload --port 8081
```

启动后:
- OpenAPI 文档:http://localhost:8081/api/docs
- 健康检查:http://localhost:8081/healthz

### 1.4 环境变量(可选)

```bash
# 仅当你想真正调用 Claude 时:
export ANTHROPIC_API_KEY=sk-ant-...

# 切换到 Postgres(可选):
export VELOCITY_API_DATABASE_URL=postgresql+psycopg://user:pass@localhost/velocity
```

不设置也能跑——后端会把 `/chat` 等 AI 端点返回 503,前端会软退化到种子内容。

### 1.5 测试

```bash
# 后端
cd apps/api && python -m pytest -q       # 应当显示 127 passed

# 前端 lint
cd ../.. && npm run lint                  # 应当显示 ~345 errors(基线)

# 前端 e2e (需要 Chromium)
npx playwright install chromium
npx playwright test                       # ~29 个用例
```

## 第 2 天 · 实战练习

强烈建议挑一个小任务真正改一遍代码,熟悉流程。建议任务(从易到难):

### 任务 A · 加一个 Skill Pack(15 分钟)

修改 `apps/api/velocity_api/seed_data.py` 的 `SKILL_PACKS` 列表,加一条新的 skill,刷新页面看是否出现在「技能中心」。学习目标:种子数据 → ORM → 前端展示链路。

### 任务 B · 加一个新的审计分类(30 分钟)

为 `Workflow` 实体的运行加审计:

1. 看 `apps/api/velocity_api/routes/automation.py`,找到 `WorkflowRun` 创建的位置。
2. 在创建后调用 `emit_audit(db, category="workflow", severity="info", action="工作流执行", ...)`。
3. 写一个 pytest 验证审计行被写入。
学习目标:`emit_audit` 的使用、pytest 模式。

### 任务 C · 给 OkrPage 加一个搜索框(60 分钟)

在 OKR 页 Objectives 列表上方加一个文本输入框,实时过滤 Objective.title。学习目标:React state、useApi 模式、不引入新 CSS 类。

### 任务 D · 把 `/route` 端点改为流式(2 小时)

参考 `chat.py` 的 `/chat/stream` 实现,把 `route.py` 的分类调用改成 SSE,前端 `RouteTester` 同步改造。学习目标:`_stream_completion` helper、SSE 前端消费、prompt caching 不被破坏。

## 三个最关键的概念

### A. 路由 → 测试 → 提交

每个新端点必须:

1. 在 `routes/X.py` 加 handler。
2. 在 `app.py` 注册 `app.include_router(X.router)`。
3. 在 `tests/test_X.py` 加至少 1 个 happy-path + 1 个 404 / 503 路径。
4. 跑 `pytest tests/test_X.py`,绿了再提交。

提交前一定跑一遍**完整套件** `pytest -q`,不要只跑改动的文件——你可能没意识到改了 schema 影响了别人。

### B. 软退化 / 种子降级 / Prompt 缓存

写新功能时永远问自己:

1. **API 完全离线时,前端是否仍能展示种子内容?**(`useApi` 钩子降级模式)
2. **未配置 `ANTHROPIC_API_KEY` 时,新端点是否返回 `503 anthropic_not_configured`?**
3. **新 system prompt 中稳定内容是否在前面、易变内容是否在后面?**(检查管理后台「Token 用量」面板的缓存命中率)

这三件事是 demo 不死、token 不烧的护城河。

### C. 不要重新发明轮子

参见 `CLAUDE.md` 的"Things to NOT do"列表:

- **不要**引入 CSS 框架(已有 token + utility 在 `index.css`)。
- **不要**引入 Redux / Zustand(`useState` + `window.dispatchEvent` 自定义事件够用)。
- **不要**引入 React Router(`setRoute` 状态机覆盖 SPA + Cmd-K 已经够)。
- **不要**批量修复 lint(项目有 ~325 条 `import React` 未使用的稳定噪音,只修自己引入的新错误)。
- **不要**引入 lucide-react(图标 map 在 `primitives.jsx`,需要新图标就贴 path)。

## 部署流程

### 触发部署

```bash
git push origin main         # 自动触发 Cloud Build trigger
```

或在 GCP Console 手动触发 trigger。`cloudbuild.yaml` 步骤:

1. Docker build(2 阶段)
2. Push 到 Container Registry
3. `gcloud run deploy velocity2 --region=us-east1`
4. Playwright e2e 跑在 `https://velocity2-...run.app`
5. 上传 HTML 报告到 GCS(失败时可看)

### 回滚

```bash
gcloud run revisions list --service=velocity2 --region=us-east1
gcloud run services update-traffic velocity2 \
  --region=us-east1 \
  --to-revisions=velocity2-PREV-REVISION=100
```

### 配置 ANTHROPIC_API_KEY

```bash
echo -n "sk-ant-..." | gcloud secrets create anthropic-api-key --data-file=-
gcloud run services update velocity2 \
  --region=us-east1 \
  --update-secrets=ANTHROPIC_API_KEY=anthropic-api-key:latest
```

## 调试常见问题

### 前端报错"Stream idle timeout"

通常是 SSE 端点没有及时刷新。看 `routes/chat.py · _stream_completion`,确认每个 token 都被 `yield`(没被 `accumulate` 吞了)。

### Pytest 偶尔失败 `test_health_json`

`/api/v1/health` 端点用全局 `SessionLocal()` 而不是测试 session,会查到 `apps/api/.data/velocity.db`。如果你在本地跑过 uvicorn 留下了脏数据:

```bash
rm -f apps/api/.data/velocity.db
pytest
```

### 缓存命中率始终为 0

可能原因:

1. prompt 长度 < 1024 tokens(Anthropic 缓存最小阈值)。
2. 每次请求 prompt 内容不一样(混入了时间戳)。
3. SDK 请求未带 `cache_control: ephemeral`。

打开管理后台「Token 用量」实时面板看「按路由」分组——如果某条路由 cache_read 始终为 0,从该路由源码倒查 system block 的稳定性。
