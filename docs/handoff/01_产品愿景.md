# 01 · 产品愿景

## 一句话定位

**Velocity OS 是一个面向中国企业的「企业知识与战略操作系统」**,把分散在文档、IM、ERP、Excel 中的企业知识与每一次战略决策绑定在一起,让 AI 不只是回答问题,而是参与完整的「认知 → 决策 → 落地 → 复盘」闭环。

## 为什么需要 Velocity

我们观察到中国大中型企业普遍存在五个结构性痛点:

1. **战略与执行断裂**:CEO 的方向与一线员工的工单之间隔着 5-7 层组织,信息每一层都被"翻译"一次,执行层难以理解决策背后的关键假设。
2. **知识沉淀低效**:90% 的关键 know-how 在群聊、个人电脑、口耳相传,新员工 6 个月才能上手。
3. **决策不可追溯**:重要选择(为什么进入 4-6 级市场?为什么选择 SC 而非 BP 主导服务网络?)结论被记下,过程被遗忘,3 个月后没人记得**当时反对的人是谁、反对理由是什么**。
4. **AI 试点孤岛**:每个部门自己接 GPT 写脚本,没有统一的知识层、权限层、审计层,合规无从谈起。
5. **跨部门协同摩擦**:工业设计部找供应链对一个材料,需要走 2 周的会议;营销做活动,服务部不知情。

## 四大核心能力

Velocity OS 围绕四个核心模块构建:

### 1. 公司知识中心 (Company Knowledge Center)

把公司全部"可被引用"的信息源统一管理:战略文档、OKR、市场报告、SOP、决策日志、供应商档案、CMF 标准等。每条知识有**域 (domain)**、**范围 (scope)**、**质量等级 (quality)**、**来源链接**。新文档通过"上传 → 入队 → 审核 → 入库"流水线进入。

### 2. 战略工作台 (Strategy Studio)

将高层战略问题(如"FY26 是否加大线上 DTC 渠道投入?")拆解为四个连续阶段:

1. **画布 (Canvas)**:把上下文知识、关联 OKR、参与 Agent 拖到空间画布上,可视化研讨准备。
2. **War Council**:多智能体辩论。7-8 个角色(财务、产品、GTM、运营、风险、供应链、组织、技术)各自基于公司知识背景给出**赞成 / 反对 / 保留**立场,可运行多轮交叉质询。
3. **战略选项**:辩论结束后,Velocity 自动综合 2-3 个候选方案(激进 / 稳健 / 渐进),每个方案带 ROI、风险、节奏、赞成 / 反对计数、推荐标记。
4. **结构化输出**:从被采纳的方案直接生成 Objective + KR + 关键项目 + 决策日志草案,一键写入 OKR 注册表。

### 3. 部门工作空间 (Department Workspace)

每个部门(集团下三级:中心 / 部门 / 团队)有自己的:

- **部门助手** (如工业设计部的"小龙虾"):被部门知识 + 公司知识同时增强。
- **部门技能包** (Skill Packs):该部门标准化的 AI 任务,如"CMF 图片识别"、"故障诊断助手"、"价格异常检测",每个技能版本化、可复用、可审计。
- **工作流模板**:多步骤的固化流程(如"创建设计简报"5 步流程,从需求 → 检索 → 推荐 → 评审 → 输出)。
- **关联项目与 OKR**:部门页直接看到该部门承担的项目与对应 KR。

### 4. 助手与治理层 (Assistants & Governance)

- **意图路由**:用户提问先经 Haiku 4.5 分类器分类到正确的部门 / 技能。
- **企业战略助手 Velocity**:CEO / VP 专属,注入全公司 OKR + 项目 + 战略知识,直接对话。
- **权限矩阵 + 审计日志**:每条 AI 调用、每次知识访问、每次规则变更都进入审计日志,可按类别 / 严重度 / 时间过滤导出 CSV。
- **管理后台**:Token 用量、模型路由策略、组织树、配额、审计五个标签。

## 北极星指标 (North Star Metrics)

衡量 Velocity OS 部署成功的六个指标:

| 指标 | 目标 | 含义 |
|---|---|---|
| 战略对齐率 | 目标 90% | 关键项目能追溯回某个 Objective + KR |
| 知识复用率 | 目标 60% | AI 回答中,引用了公司已有知识源的比例 |
| 决策可追溯率 | 100% | 重要决策都进入决策日志,带假设 + 反对意见 |
| 部门助手覆盖率 | 目标 80% | 已配置部门助手的部门 / 总部门 |
| 项目健康度可视化 | 100% | 关键项目都有健康灯 + 风险条目 |
| AI 缓存命中率 | 目标 ≥ 60% | Anthropic prompt caching 的命中比例,反映 prompt 工程质量 |

## 设计原则

1. **知识必须可追溯**:任何 AI 回答必须能下钻到原始知识源。
2. **部门必须可配置**:不是写死的"工业设计部",而是任意三级组织树都能挂助手 / 技能 / 工作流。
3. **战略必须能落地**:辩论的产出要能直接变成 OKR / 项目 / 决策日志草案。
4. **缓存与命中率优先**:prompt 设计始终把稳定上下文(公司 / 部门 / 技能)放在缓存前缀,把易变上下文放在尾部。
5. **本地优先,云端可选**:开发零配置(SQLite + 默认种子),云端通过单一容器部署。

## 与早期版本的关系

本愿景是基于 v1 (Cognitive Factory)、v2 (Knowledge & Strategy OS) 之上的实施落地版本。相对 v2 PRD,实际交付的差异:

- **多智能体辩论真正落地**:v2 仅描述了 War Council 的概念,本版本通过 Sonnet 4.6 多次调用 + 流式 SSE 真正实现了 7 个 Agent 的角色扮演辩论。
- **结构化输出从草稿变为应用**:战略选项 + 结构化输出现在可以一键写入真实 OKR / 项目 / 决策表。
- **prompt caching 作为头等关注点**:管理后台有专门面板展示缓存命中率。
