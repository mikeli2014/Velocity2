# 08 · 待办路线图(What's Left)

本文档列出当前 Velocity OS 在主干分支(`main` @ `db6b186`)状态下尚未完成的工作。按优先级分四档:

- **P0 · 上生产前必做**:不解决直接影响真实客户使用。
- **P1 · 近期高价值**:1-2 个迭代内做完,显著提升体验或可运维性。
- **P2 · 中期排期**:有明确价值,但当前替代方案能用。
- **P3 · 未来探索**:思路尚未收敛,需要先讨论再排期。

每条待办都标注:**估工**(idealized 工程师人天)、**依赖**、**主要影响文件**。

## P0 · 上生产前必做

### P0-1 · 切换 Cloud Run 的数据库到 Cloud SQL Postgres

**问题**:当前 Cloud Run 仍用 SQLite,实例冷启动后用户产生的 OKR / 项目 / 决策 / 审计行全部丢失,无法承载真实用户。

**做法**:
1. `gcloud sql instances create velocity-db --tier=db-f1-micro --region=us-east1`
2. `gcloud sql databases create velocity --instance=velocity-db`
3. `gcloud run services update velocity2 --add-cloudsql-instances=... --update-env-vars=VELOCITY_API_DATABASE_URL=postgresql+psycopg://...`
4. **关键测试**:在切换后立即跑 Playwright e2e + 手动跨冷启动验证 OKR 创建持久化。

**估工**:0.5 人天(代码已就绪,纯运维)。
**依赖**:GCP 项目权限;预算审批(`db-f1-micro` 大约 $7/月)。
**影响文件**:`cloudbuild.yaml`(可选)、Cloud Run service config、Cloud SQL 实例。

### P0-2 · 引入 Alembic 迁移

**问题**:当前 schema 通过 `Base.metadata.create_all()` 在启动时创建。一旦切到 Postgres + 真实数据,改 schema 不能再通过删表重建。

**做法**:
- 接入 `alembic` 包,生成 baseline 迁移(对应当前 19 张表)。
- 在 `app.py` lifespan 中 `command.upgrade(config, "head")` 替换 `create_all`。
- 文档化"修改 model 后的标准动作":`alembic revision --autogenerate -m "..."` → 检查迁移 → 提交。

**估工**:1.5 人天。
**依赖**:P0-1 切到 Postgres(SQLite 上跑迁移意义不大)。
**影响文件**:`apps/api/alembic/`(新建)、`apps/api/velocity_api/app.py`、`apps/api/CLAUDE.md`。

### P0-3 · 引入真实 OAuth(替代 X-User-Id 头)

**问题**:`X-User-Id` 是开放头,任意用户都能写入审计的 `actor` 字段——伪装成 CEO 不需要任何成本。

**做法**:
- 选择身份提供方(Google Workspace / 企业微信 / 钉钉 OAuth)。
- 在 `velocity_api/auth.py` 新增 `current_user_oauth` 依赖,验证 JWT。
- 保留 `current_user_xheader` 作为 dev / e2e 测试模式。
- 通过环境变量 `VELOCITY_API_AUTH_MODE=header|oauth` 切换。
- 前端添加登录页 + token 存储 + Authorization 头透传。

**估工**:5 人天(含前端登录流)。
**依赖**:确定身份提供方。
**影响文件**:`apps/api/velocity_api/auth.py`、`src/lib/api.js`、新建 `src/pages/LoginPage.jsx`。

## P1 · 近期高价值

### P1-1 · RAG(pgvector + 真实文档检索)

**问题**:助手回复目前只能引用 `KnowledgeSource.summary` 短摘要,没有全文检索。Strategy Studio 引用的 `ks-N` 也仅是元数据。

**做法**(详细方案见 `04_知识管理.md` §8 + 此前的设计 brainstorm):
1. 选择 embedding 服务:Voyage AI(推荐,Anthropic 合作伙伴) vs 本地 `sentence-transformers`。
2. 新增 `KnowledgeChunk` ORM 表(`source_id, idx, text, embedding JSON, tokens`)。
3. 新增 `POST /knowledge-sources/{id}/generate` 端点:Claude 生成 ~1500 token 全文 → 600 字符切块 → embed → 持久化。
4. 新增 `POST /knowledge/search` 端点:embed query → 余弦 top-K(numpy in-process,< 5K chunks 性能足够)→ 返回 `{sourceId, snippet, score}`。
5. `/chat{,/stream}` 新增 `retrieve: true` 标志,命中时把 top chunks 作为 volatile system block 注入。
6. 前端 `AssistantChat` 加"📚 检索知识库"开关。

**估工**:5-7 人天。
**依赖**:决定 embedding 服务 + Voyage 或同类 API key。
**影响文件**:`models.py`、`schemas.py`、新建 `routes/knowledge_search.py` + `services/embedding.py`、`chat.py`、`AssistantChat`。

### P1-2 · Strategy Canvas 真实数据驱动

**问题**:`StrategyCanvas` 的 OKR / Project / 风险节点位置仍按 sq-1 写死;新建战略问题进画布看到的内圈外圈只是默认排布。

**做法**:
- 让 Canvas 读 `question.context`(知识源 ID 列表)和 `question.okrs`(OKR 编号),按 ID 找实体并放置。
- 工具栏「+ 添加上下文」「+ 添加 Agent」实际写回 `question.context` / `question.agents`(`PATCH /strategy-questions/{id}`,新端点)。
- 拖拽位置持久化(可选,需新表 `CanvasLayout`)。

**估工**:3 人天(不含拖拽)。
**依赖**:无。
**影响文件**:`StrategyPage.jsx · StrategyCanvas`、`routes/catalog.py`(加 PATCH)、`schemas.py`。

### P1-3 · 工作流真实执行引擎

**问题**:`RunDialog` 当前的 Workflow 运行只是步骤动画 + 单次 `/chat` 调用;`Workflow.steps[]` 的 `kind`(human / system / skill / ai / approval)字段没有对应的 runner。

**做法**:
- 新增 `POST /workflows/{id}/runs/start` SSE 端点。
- 后端按 step 顺序执行:
  - `kind=ai`:调 Sonnet,绑定 step 输入 → 输出。
  - `kind=skill`:加载 SkillPack 的输入输出契约 + system prompt。
  - `kind=human` / `approval`:暂停,等待 `POST /runs/{run_id}/steps/{step_id}/resolve`。
- 前端 `RunDialog` 订阅 SSE,逐步推进 UI。

**估工**:8 人天(状态机复杂)。
**依赖**:无。
**影响文件**:新建 `routes/workflow_runs.py`、`models.py` 加 `WorkflowStepRun`。

### P1-4 · 配额真实强制 + 告警

**问题**:管理后台「配额与预算」只显示静态数据;真实 Token 用量和配额不挂钩。

**做法**:
- 新增 `DeptQuota` 表(部门、月度上限、当前消耗)。
- `record_llm_call` 触发器:命中部门标签后 `quota.consumed_tokens += ...`。
- 超限时 chat / debate 端点返回 `429 quota_exceeded`。
- 阈值告警:消耗 > 80% 时发邮件 / 钉钉通知。

**估工**:3 人天。
**依赖**:无。
**影响文件**:`models.py`、`telemetry.py`、所有 AI 路由、`AdminPage` 配额面板。

### P1-5 · 知识图谱真实化

**问题**:`KnowledgePage · 知识图谱` 标签仍是静态 SVG。

**做法**:
- 后端新增 `GET /knowledge/graph` 返回 `{nodes: [{id, label, domain, type}], edges: [{source, target, kind}]}`。
- 边的来源:`linked_projects`、`linked_decisions`、域成员关系、引用关系(等 RAG 上线后)。
- 前端用 force-directed layout(可选 `react-force-graph` 或自研 SVG)。

**估工**:4 人天。
**依赖**:RAG(P1-1)上线后图谱才"长肉"。
**影响文件**:新建 `routes/knowledge_graph.py`、`KnowledgePage`。

## P2 · 中期排期

### P2-1 · 文档解析流水线(PDF / DOCX / PPT → 文本)

当前 IngestQueue 状态机里 `parsing → embedding → tagging` 是占位,真实没有解析逻辑。需要接入 `pypdf` / `python-docx` / `python-pptx` 或外部 service(Azure Document Intelligence)。**估工**:3 人天(简单格式) / 2-3 周(扫描 PDF + 表格)。

### P2-2 · Cmd-K 全文搜索

当前 `CommandPalette` 只按 title 前缀。**做法**:
- Postgres `tsvector` GIN 索引 + 中文分词(`zhparser` 扩展),或接入 Elasticsearch / Meilisearch。
- 端点 `GET /search?q=...&types=okr,project,knowledge,decision`。
**估工**:2-4 人天。

### P2-3 · LLMCall 用量按日 roll-up

当前 `/llm-usage/summary` 每次扫全表;数据量大后会慢。**做法**:
- 新增 `LLMUsageDailyRoll` 表,定时任务(Cloud Scheduler + Cloud Run job)按日聚合。
- summary 端点优先读 roll-up,只有当天的查 raw。
**估工**:2 人天。

### P2-4 · 路由分类器接入 Cmd-K 失败回落

当 Cmd-K 找不到匹配项时,自动调 `/route` 把查询分类到部门并提示"打开 X 部门助手 → ?"**估工**:1.5 人天。

### P2-5 · Activity 反向连接更多事件源

当前 HomePage Activity feed 主要来自审计日志的写事件;还应包括:
- 战略问题创建 / 状态变更
- 工作流运行
- KR Check-in 提交
**估工**:2 人天(需在多处插桩)。

### P2-6 · Mobile 端 Playwright 测试覆盖

CSS 已加移动端 @media,但 Playwright 没有 mobile viewport project,实际能否用没有自动化验证。**做法**:在 `playwright.config.js` 加 `iPhone 12` 项目,把 features.spec 复用到 mobile。**估工**:1 人天。

### P2-7 · 审计日志 TTL / 归档

审计行无限累积,长期会爆。**做法**:Cloud Scheduler 按月归档到 GCS,DB 只留 90 天。**估工**:1.5 人天。

### P2-8 · 演示数据"重置"端点

老板想 demo 之前一键清空所有用户产生的数据回到种子。**做法**:`POST /admin/reset-to-seed` + 管理后台一个红色按钮(带确认弹窗)。**估工**:0.5 人天。

## P3 · 未来探索

### P3-1 · Claude Computer Use 接入

让 Velocity 助手在用户授权下直接操作其他系统(SAP / 钉钉 / 企业微信)。需要 Anthropic computer-use beta + 工具白名单 + 录屏审计。**估工**:未评估,概念验证至少 5 人天。

### P3-2 · Self-Reflection / Disagreement Loops

战略辩论第二步:让 Sonnet 对自己上一轮的发言做 self-critique,产出"我之前说的什么是错的"修正稿。可能显著提升综合摘要质量。需先做评估,看 token 成本是否值得。

### P3-3 · 多语言支持

当前 UI 全中文。如果要海外客户,需:i18n 框架(`react-intl`)+ 全量字符串提取 + 翻译 + 系统 prompt 多语言切换。**估工**:大,15+ 人天。

### P3-4 · 移动原生 App

React Native 版本。当前响应式 Web 在移动端可用,但有原生 IM 集成(企业微信深度链接、推送)需求才值得做。**估工**:无评估,从零至少 1 人月。

### P3-5 · 多租户 / SaaS 化

把单部署改造为多公司租户。`Company` 表已经是单行,需要加 `tenant_id` 到所有表 + 中间件根据子域 / 租户头切换。**估工**:大,2-3 人月,涉及全部 CRUD 端点。
