# 05 · 智能体工作流

本文档描述 Velocity OS 中三类 AI 智能体如何工作、prompt 如何组织、何时缓存、如何审计。

## 1. 三种智能体形态

| 形态 | 模型 | 调用模式 | 用例 |
|---|---|---|---|
| **对话助手** (部门 / 公司) | Sonnet 4.6 | 单调用,流式 | 用户与助手 1:1 对话 |
| **多 Agent 辩论** (War Council) | Sonnet 4.6 | 一轮多次调用,流式 | 7-8 个角色就一个战略问题辩论 |
| **路由分类器** | Haiku 4.5 | 单调用,非流式 | 把用户提问归类到部门规则 |

下面分别说明。

## 2. 对话助手 (Department / Company Assistant)

### 2.1 流程

```
用户输入 ── apiStream("/chat/stream", {messages, deptId})
                        │
                        ▼
        FastAPI · routes/chat.py · /chat/stream
                        │
            构建 system_blocks ──┬── [Company Block] (cache_control)
                                 ├── [Dept Block]    (cache_control,可选)
                                 ├── [Skill Block]   (cache_control,可选)
                                 └── [Volatile suffix]    (不缓存)
                        │
                        ▼
        Anthropic SDK · client.messages.stream(...)
                        │
            循环 ── stream.text_stream
                        │
            yield SSE  data: {"type":"text","text":"片段"}
                        │
                        ▼
            stream.get_final_message() ── 完成
                        │
            yield SSE  data: {"type":"done","model","usage"}
                        │
                        ▼
            record_llm_call(db, route="chat.stream", ...)
                        │
                        ▼
            前端 onEvent 累积 buffered text 到对应消息气泡
```

### 2.2 Prompt 设计要点

- **Company Block** 包含 `Company.name + name_en + tagline + industry + fiscal_year + revenue`,以及 PRD §5.1 描述的 `focus_areas` / `competitors` / `terminology` / `context_prompt` 字段(种子默认空,可由"公司档案"编辑器填充)。
- **Dept Block** 注入部门负责人、人员规模、当前部门助手名(`小龙虾`/`小服`等)以提示语气。
- **Skill Block** 仅在 `RunDialog` 调用时注入,描述当前正在执行的技能输入 / 输出预期。
- **Volatile suffix** 用于 per-request 限制,如"请用一句话回答"——不缓存。

### 2.3 软退化

未配置 `ANTHROPIC_API_KEY`:
- 后端立即返回 `503 anthropic_not_configured`,**不**记录 LLMCall(因为没有真实调用)。
- 前端 `AssistantChat` 检测到该 slug,展示一段中文"助手未配置"的占位回复,保证演示不白屏。

## 3. 多 Agent 辩论 (War Council)

### 3.1 角色清单

种子的 8 个 Agent 角色(`Agents` 表):

| ID | 名称 | 关注点 | 颜色 / 图标 |
|---|---|---|---|
| ag-finance | 财务视角 | ROI / 现金流 / 投入产出比 | 橙 / Coins |
| ag-product | 产品视角 | 用户价值 / 品类机会 / 路线图 | 靛 / Package |
| ag-gtm | GTM 视角 | 渠道 / 节奏 / 竞争反应 | 粉 / Megaphone |
| ag-ops | 运营视角 | 执行节奏 / 资源 / 流程 | 绿 / Activity |
| ag-risk | 风险视角 | 合规 / 安全 / 反对意见 | 红 / Shield |
| ag-tech | 技术视角 | 可行性 / 架构 / 投入 | 天蓝 / Cpu |
| ag-supply | 供应链视角 | 产能 / 库存 / 供应商 | 紫 / Truck |
| ag-org | 组织视角 | 组织 / 人才 / 文化 | 紫罗兰 / Users |

每个角色有自己的"立场"标签,Agent 被要求以 `[赞成]`/`[反对]`/`[保留]` 开头作答。

### 3.2 一轮辩论的执行

`POST /api/v1/strategy-questions/{id}/debate/round/stream`(SSE):

```python
for agent in selected_agents:
    yield SSE data: {"type":"agent_start","agentId": agent.id}

    # 每个 agent 各自的系统 prompt:
    system_blocks = [
        {company_block,    cache_control: ephemeral},  # 全局共享
        {question_block,   cache_control: ephemeral},  # 全局共享(含证据知识源)
        {agent_persona,    cache_control: ephemeral},  # 每个 agent 的人设(差异点)
    ]
    # 用户回合:当前轮次提示 + 已有发言摘要(volatile)
    user_msg = f"现在是第 {round_no} 轮:{round_kind}\n\n已有发言:\n{transcript}\n\n以「{agent.name}」身份发言。"

    with client.messages.stream(...) as stream:
        for token in stream.text_stream:
            yield SSE data: {"type":"text","agentId": agent.id,"text": token}
        final = stream.get_final_message()

    # 解析立场前缀
    stance, body = _parse_stance(buffered)
    cited = _scrape_source_ids(body, valid_source_ids)

    # 持久化 DebateMessage 行
    row = DebateMessage(stance=stance, text=body, sources=cited, ...)
    db.add(row); db.flush()

    yield SSE data: {"type":"agent_done","agentId": agent.id,"message": _to_out(row)}

    record_llm_call(db, route="debate.round.stream", model, usage)
```

7 个 Agent 顺序调用,但每个 Agent 都对前缀缓存命中(Company + Question 块字节稳定),只有人设和最后的 user turn 是新的。这就是为什么管理后台能看到 60-80% 的缓存命中率。

### 3.3 立场解析

```python
_STANCE_PATTERNS = [
    (re.compile(r"^\s*\[\s*赞成\s*\]"), "pro"),
    (re.compile(r"^\s*\[\s*反对\s*\]"), "con"),
    (re.compile(r"^\s*\[\s*保留\s*\]"), "concern"),
]
```

匹配后 stripping 前缀,`text` 字段只存正文。如果 Agent 没按规则开头,默认 `concern`。

### 3.4 综合摘要

辩论运行 N 轮后,用户点击「重新生成摘要」→ `POST /debate/synthesis/stream`:

- 系统 prompt:`[Company] [synthesis-instructions]`(都缓存)。
- 用户回合:完整辩论记录 + 立场计数(volatile)。
- 输出:核心张力 + 各阵营观点 + 一条可执行的下一步(3-4 句中文)。
- SSE 第一个事件是 `data: {"type":"counts","pro":3,"con":1,"concern":3}`,前端立即更新立场计数 pill。

## 4. 路由分类器 (Haiku 4.5)

### 4.1 设计

只用 1 个 Haiku 调用,输入用户原始提问,输出严格格式:

```
rule_id=rt-1;confidence=0.92;rationale=匹配 CMF 工艺关键词
```

或 `rule_id=none` 表示无匹配。

### 4.2 缓存友好

```python
system_blocks = [
    {company_block, cache_control: ephemeral},      # 总是相同
    {rule_catalog,  cache_control: ephemeral},      # 规则列表(增删时变化)
]
user_msg = payload.text   # 真实查询(不缓存)
```

只要规则列表不变,1000 次路由调用都共享同一个缓存前缀,只付 user_msg 的 tokens。

### 4.3 何时调用

- **现状**:`AssistantsPage` 的「路由测试」面板,主要用于 IT 调试规则。
- **路线**:Phase 3 把 Cmd-K 全局搜索的"未命中条目时"自动跳路由分类,以便用户不知道找谁问的时候被自动指引到正确的部门助手。

## 5. Skill Pack 的位置

Skill Pack(`SkillPacks` 表)是当前系统中"未来可执行"的占位:每个技能定义了 input / output / 步骤,但 `RunDialog` 实际调用 `/chat` 时,只是把 Skill 信息注入 system prompt,并非真正按步骤分步执行。

未来若要"真正的 Workflow Engine":

- 让 `Workflow.steps` 中每一步都有 `kind`(skill / human / approval / ai-step)。
- 后端新增 `POST /workflows/{id}/runs` 端点,按步骤逐个执行,流式 SSE 反馈。
- 现在的 `WorkflowRun` 表已经有 `status` / `output` 字段,只是没有真实 runner。

## 6. 共同的可观测性

所有 AI 调用都通过 `velocity_api/telemetry.py · record_llm_call` 落入 `LLMCall` 表:

```
LLMCall(id, route, model, input_tokens, output_tokens,
        cache_read_input_tokens, cache_creation_input_tokens,
        latency_ms, status, error_detail, created_at)
```

`route` 字段当前的 slug:
- `chat`、`chat.stream`
- `debate.round`、`debate.round.stream`
- `debate.synthesis`、`debate.synthesis.stream`
- `strategy.options`、`strategy.structured_output`
- `route`

管理后台 → Token 用量 → 「实时 AI 用量」面板:

- 累计调用 / 输入 token / 输出 token / 缓存命中率 4 个 KPI。
- 按模型 / 按路由两个汇总表。
- 最近 N 次调用明细(状态、延迟、token、缓存读)。

刷新按钮触发 `useApi.refresh()`。

## 7. Prompt 工程的纪律

为保持缓存命中率,在 `chat.py` / `debate.py` 修改 prompt 时遵循:

1. **稳定内容前置**:Company → Department → Skill 顺序写入 system,且每块的字节内容必须可重现(基于数据库行,不是当前时间)。
2. **易变内容后置**:用户输入 / 历史摘要 / 当前轮次提示放在 `messages` 数组,**不要**塞进 system block。
3. **Cache-control 标记**:稳定块都加 `{"type": "ephemeral"}`,易变块**不要**加。
4. **修改 prompt 必须看 LLMCall 行**:打开管理后台用量面板,调用前后对比 `cache_read_input_tokens` 是否仍上升。
5. **Adaptive thinking 暂未启用**:目前用 `messages.create` 默认行为;未来引入 `thinking: {type: "adaptive"}` 时同步 SDK 升级。
