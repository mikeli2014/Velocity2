## 三、生产部署上的发现(本轮新增坑)

### A. 依赖飘移(已修)

`anthropic 0.39.0` 调用 `httpx.Client(proxies=...)`,但 `httpx 0.28+` 移除了该 kwarg。`requirements.txt` 没有 pin httpx,Docker 重建时拉到最新 0.28+,导致 `Anthropic(api_key=...)` 构造立即 TypeError → /chat /route 全部 500。

修复:`httpx>=0.27,<0.28`。详见 `3f0694d` commit message。

**教训**:依赖之间的兼容性边界需要显式表达。建议:每个间接依赖只要 transitive,且 SDK 内部直接构造它(如 anthropic→httpx),都应当在 requirements.txt 中 pin 上下界。

### B. Cloud Run 路径反射(已修)

Cloud Run legacy URL `https://velocity2-{PROJECT_NUMBER}.{REGION}.run.app` 会把 `/healthz` 路径反射给 GFE 内部探针,直接返回 Google 的 404 页(带 Google logo),不会到达容器。

经典 URL 格式 `https://velocity2-{HASH}-{REGION_SHORT}.a.run.app` 不拦截。

修复:测试改用 `/api/v1/health`(应用层),不依赖底层路径名。

### C. 安全债(待用户行动)

会话过程中两次直接把 `ANTHROPIC_API_KEY` 贴进 chat。两把 key 都已记入对话日志。**等用户去 console.anthropic.com 撤销并新建 key。**

后端 Secret Manager 的接入流程已经验证可用:

```
gcloud secrets create anthropic-api-key --data-file=newkey.txt
gcloud secrets add-iam-policy-binding anthropic-api-key \
  --member=serviceAccount:802659517720-compute@developer.gserviceaccount.com \
  --role=roles/secretmanager.secretAccessor
gcloud run services update velocity2 --region=us-east1 \
  --set-secrets=ANTHROPIC_API_KEY=anthropic-api-key:latest
```

### D. Anthropic 余额耗尽(待用户行动)

`/chat` 现在能正常调到 Anthropic,但账户余额不足,被 400 拒绝:
