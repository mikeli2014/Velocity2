# 04 · 知识管理

本文档描述 Velocity OS 的知识体系:从一份原始文档进入系统,到被 AI 引用回答问题的完整路径,以及知识官 / IT 运营的日常职责。

## 1. 知识层次

Velocity OS 把企业知识分成三个粒度,从粗到细:

```
公司级知识 (Company-level)
   │  战略文档、年度 OKR、组织架构、合规手册等全员可见
   │
   ▼
部门级知识 (Department-level)
   │  CMF 标准、SOP、客户档案、供应商能力库等;部门成员可见
   │
   ▼
项目级知识 (Project-level)
   会议纪要、设计简报、代码评审等;项目参与人可见
```

每条知识源 (`KnowledgeSource`) 都有 `scope` 字段,值 `公司 / 部门 / 项目`,决定可见性。

## 2. 知识域 (Knowledge Domain)

知识域是公司维度的"信息池子",每个域承载一类知识。当前种子的 8 个域:

| ID | 名称 | 范围 | 健康度 | 覆盖率 | 标签 |
|---|---|---|---|---|---|
| kd-cmf | CMF 与材料 | company | ok | 78% | CMF / 材料 / 工艺 |
| kd-strategy | 战略与 OKR | company | ok | 91% | 战略 / OKR / 决策 |
| kd-channel | 渠道运营 (BP/SC/SA) | company | warn | 64% | 渠道 / KA / 县域 |
| kd-service | 服务与售后 | company | ok | 82% | 工单 / SOP / NPS |
| kd-supply | 供应链与产能 | company | ok | 73% | 供应商 / BOM / 排产 |
| kd-market | 市场与竞品 | company | warn | 58% | 奥维 / 竞品 / 价格带 |
| kd-product | 产品规划 | company | ok | 85% | 路线图 / 用户画像 |
| kd-finance | 财务与配额 | company | not-started | 0% | 预算 / ROI / 现金流 |

知识域用于:

- 控制 Skill Pack / 助手能引用哪些知识(`workflow.linked_domains` 字段)。
- 知识中心首页的覆盖率仪表板。
- 工作流 × 知识域的二维矩阵(WorkflowsPage 标签页)。

## 3. 知识源 (Knowledge Source)

单条知识(一份 PDF、一个 URL、一段文档)。字段:

| 字段 | 说明 |
|---|---|
| id | `ks-1`、`ks-2`...,kebab-case |
| title | 显示标题 |
| type | `PDF` / `URL` / `DB` / `MD` |
| scope | `公司` / `部门` / `项目` |
| quality | `verified`(经审核) / `draft`(待审) / `flagged`(标记) |
| owner | 负责人显示名 |
| size | 显示用("8.4MB"/"38 页") |
| summary / excerpt | 自动摘要 / 节选(供 RAG 用) |
| tags | 标签数组 |
| pages, lang | 文档元数据 |
| uploaded_by | 上传者(由 `X-User-Id` 写入) |
| embeddings | 向量数(Phase 3 启用) |
| linked_projects, linked_decisions | 反向关联 |

种子数据中的 8 条知识源(按公司类型):FY26 集团战略、CMF 中台手册、奥维 Q4 报告、BP/SC/SA 操作手册、供应商能力库、县域服务网络说明等。

## 4. 入库流水线

```
                   1. 上传 / 抓取
        ┌────────┐  PDF / URL / DB
        │用户 / IT│ ─────────────────┐
        └────────┘                   │
                                     ▼
                              ┌──────────────┐
                              │ IngestQueue  │ ◀── 状态机
                              └──────┬───────┘
                                     │
   ┌──── fetching ──── parsing ──── embedding ──── tagging ──── review
   │                                                                 │
   │  失败时 state=failed,带错误信息;管理员可点击 retry          │
   │                                                                 ▼
   │                                                            ┌────┴────┐
   │                                                            │  审核员  │
   │                                                            └────┬────┘
   │                                                                 │
   │                                                       拒绝 ◀──┴──▶ 入库
   │                                                       (rejected)   │
   │                                                                    ▼
   │                                                            ┌──────────────┐
   │                                                            │KnowledgeSource│
   │                                                            └──────────────┘
```

**关键端点**:`POST /api/v1/ingest-queue/{id}/approve` 把 review 状态的队列项变成正式 `KnowledgeSource`,审计日志同时写入 `category=knowledge / action=知识源入库`。

## 5. AI 与知识的连接

### 5.1 助手对话(Department Page)

部门助手(`AssistantChat` 组件 → `/api/v1/chat/stream`)的系统 prompt 由三块组成:

```
[Company Block] ── 公司事实(focus_areas、competitors、术语)── 缓存
[Dept Block]    ── 部门人设、负责人、知识量、技能列表    ── 缓存
[Skill Block]   ── 当前正在调用的技能(可选)             ── 缓存
[Volatile]      ── 单次 system_suffix(可选)             ── 不缓存
```

回答中如果包含 `ks-N` 形式的源 ID,前端会渲染成可点击的 chip,跳转到知识中心查看原文。

### 5.2 战略辩论(War Council)

每个 Agent 调用的系统 prompt:

```
[Company Block]  ── 缓存(7 个 Agent 共享)
[Question Block] ── 含 question.context 引用的知识源全文摘要 ── 缓存
[Agent Persona]  ── 当前 Agent 的角色定位、风格、关注点    ── 缓存(每 Agent 独立)
[User Turn]      ── 当前轮次提示 + 历史辩论摘要            ── 不缓存
```

辩论时,Agent 被要求**强引用**(`引用 ks-4`),便于追溯证据。

### 5.3 路由分类(Haiku)

所有启用的 `RoutingRule` 序列化进单一缓存 system block:

```
可选路由规则(按优先级):
- id=rt-1; intent=CMF / 材料 / 工艺 / 色彩 / 表面; dept=industrial-design; skill=sp-mat-search; ...
- id=rt-2; intent=...
...
```

Haiku 4.5 输出严格格式 `rule_id=...;confidence=...;rationale=...`,每次调用之间规则列表稳定 → 缓存命中。

## 6. 质量控制

| 措施 | 实现位置 |
|---|---|
| 入库前 review 状态闸口 | `IngestQueue.state == "review"` |
| 知识源质量等级 | `KnowledgeSource.quality` (verified/draft/flagged) |
| 反向引用计数 | `linked_projects` / `linked_decisions` 长度 |
| 审计日志 | 每次新增 / 更新 / 删除知识源都有 `category=knowledge` 行 |
| 实时使用量 | `KnowledgeSource.uses` 计数(Phase 3 与 LLM 调用绑定) |

## 7. 知识官的日常工作

每周 / 每天的典型动线:

1. **早晨**:进入「公司知识中心 → 上传队列」,处理昨天的 review 状态文件——预览原文 → 决定入库 / 拒绝 / 退回补全标签。
2. **每周**:看「知识源」列表,按"最近更新"排序,把 30 天没人引用的知识标 `flagged`(质量待复核)。
3. **每月**:看「知识图谱」标签下的覆盖率仪表板,识别低覆盖域(如 `kd-finance` 0%),督促对应部门补充。
4. **战略研讨前**:确保被 `StrategyQuestion.context` 引用的源都是 `verified` 等级,否则 War Council 引用会被打折。
